# IEEE 802.11 WiFi 5 (802.11ac / VHT) — Current INET Model Overview

Baseline study of what the existing INET 802.11 model supports, up to and
including WiFi 5 (802.11ac / VHT). This is the detailed baseline referenced by §1
of the [next-generation implementation plan](wifi6-wifi7-802.11ax-be.md) — it
documents the structure that extension should reuse, and the gaps that already
exist at WiFi 5.

Scope: studied `src/inet/physicallayer/wireless/ieee80211/` (PHY),
`src/inet/linklayer/ieee80211/` (MAC/mgmt/mib/llc/portal), the examples/
showcases, and the User's Guide chapter `doc/src/users-guide/ch-80211.rst`.

WiFi marketing ↔ amendment ↔ INET mode naming:

| Marketing | Amendment | PHY family | INET status |
|-----------|-----------|------------|-------------|
| (legacy)  | 802.11-1997 | DSSS / FHSS / IR | present |
| —         | 802.11a   | OFDM       | present (`Ieee80211OfdmMode`) |
| —         | 802.11b   | HR-DSSS / CCK | present (`Ieee80211HrDsssMode`) |
| —         | 802.11g   | ERP-OFDM / DSSS-OFDM | present |
| —         | 802.11p   | OFDM (10 MHz, 5.9 GHz) | present |
| WiFi 4    | 802.11n   | HT         | present (`Ieee80211HtMode`) |
| WiFi 5    | 802.11ac  | VHT        | present (`Ieee80211VhtMode`) — **current ceiling** |

INET's own self-assessment, from `linklayer/ieee80211/mac/Ieee80211Mac.ned:33`:
> "Support for 802.11a/b/g/e/p, as well as partial/incomplete support for later
> WiFi standards such as n and ac."

The one-line summary: **the VHT PHY rate set is implemented in full, but 11ac is
"partial/incomplete" above the PHY** — there is no VHT capability negotiation, no
MU-MIMO / beamforming, no real MIMO channel, and no security.

---

## 1. Officially supported standards — the `opMode` enum

From `physicallayer/wireless/ieee80211/packetlevel/Ieee80211Radio.ned:28`:

```
string opMode @enum("a", "b", "g(erp)", "g(mixed)", "n(mixed-2.4Ghz)", "p", "ac")
              = default("g(mixed)");
```

| opMode | Standard | Notes |
|--------|----------|-------|
| `a` | 802.11a | 5 GHz OFDM |
| `b` | 802.11b | 2.4 GHz HR-DSSS / CCK |
| `g(erp)` / `g(mixed)` | 802.11g | ERP-only vs. 802.11b-compatible |
| `n(mixed-2.4Ghz)` | 802.11n (HT) | **2.4 GHz only** — there is no 5 GHz `n` mode set |
| `p` | 802.11p | 5.9 GHz vehicular / WAVE |
| `ac` | **802.11ac (VHT) — WiFi 5** | 5 GHz |
| — | later amendments (after 802.11ac) | **not present at all** |

## 2. Physical layer

All PHY modes live in `physicallayer/wireless/ieee80211/mode/`. Each 802.11
generation through `ac` has a mode class; there is nothing beyond `ac`. The mode
hierarchy (`IIeee80211Mode` → preamble / signal / data submodes + MCS table +
bandwidth + spatial streams + guard interval) is clean and is the template the
next-generation extension should follow.

### 2.1 VHT (802.11ac / WiFi 5) — fully populated rate set

Defined in `mode/Ieee80211VhtMode.{h,cc}`, `mode/Ieee80211VhtCode.{h,cc}`, and
registered as the `"ac"` mode set in `mode/Ieee80211ModeSet.cc`.

| Capability | Status | Detail |
|------------|:------:|--------|
| Bandwidths 20 / 40 / 80 / **160 MHz** | ✅ | All four in the `"ac"` mode set (160 MHz incl., e.g. `Ieee80211ModeSet.cc:377-396`); bands in `mode/Ieee80211Band.cc` (`5 GHz (20/40/80/160 MHz)`) |
| 80+80 MHz (non-contiguous) | ❌ | not modeled |
| MCS 0–9 (incl. **256-QAM**) | ✅ | per-spec validity respected (e.g. MCS9 @ 20 MHz / NSS1 marked "No valid") |
| Spatial streams NSS 1–8 | ✅ | full MCS × NSS × BW tables in `Ieee80211VhtMode.cc` |
| Short / long guard interval (400 / 800 ns) | ✅ | both GIs registered in the mode set |
| Convolutional FEC (1/2, 2/3, 3/4, 5/6) | ✅ | `Ieee80211VhtCode` |
| **LDPC FEC** | ❌ | `// TODO LDPC codes` in `Ieee80211VhtCode.h` |
| **DL MU-MIMO** | ❌ | not modeled |
| **Transmit beamforming / sounding (NDP)** | ❌ | not modeled |
| **STBC** | ❌ | explicit comment: "we assume STBC is not used" |

**Important caveat — MIMO is a rate multiplier, not a channel.** NSS increases the
*computed bitrate* (and thus shortens airtime), but there is no MIMO channel
matrix, per-stream fading, spatial-multiplexing capacity, or beamforming gain.
Setting `numAntennas = 8` with a high NSS affects throughput/airtime accounting,
not link-reliability physics.

### 2.2 HT (802.11n) — for comparison

`mode/Ieee80211HtMode.{h,cc}`, `mode/Ieee80211HtCode.{h,cc}`. MCS 0–31 (up to
64-QAM), 20/40 MHz, NSS up to 4, short/long GI, convolutional FEC. Same submode
structure as VHT. Exposed only as `n(mixed-2.4Ghz)` (2.4 GHz).

### 2.3 Error models and abstraction levels

- **Packet-level** (default, fast): NIST and YANS error models in
  `packetlevel/errormodel/` — both handle OFDM / HT / VHT.
- **Bit-level** (detailed): scrambler, convolutional coder, HT/VHT interleaver,
  OFDM (de)modulation in `bitlevel/`. VHT interleaving present; no LDPC.
- **Radio types:** `Ieee80211ScalarRadio`, `Ieee80211DimensionalRadio`,
  `Ieee80211UnitDiskRadio`, plus a layered bit-level OFDM radio. (The User's Guide
  documents only the first three.)

## 3. MAC layer

`linklayer/ieee80211/mac/` — the strongest, most mature part of the model, shared
across every `opMode`. Deliberately modular: coordination functions, channel
access, MAC data services, frame sequences, and policies are all replaceable
submodules, so research swaps one component without rewriting the MAC.

| Area | Status |
|------|--------|
| **DCF** (CSMA/CA basic access) | ✅ full |
| **HCF / EDCA** (QoS; 4 ACs BK/BE/VI/VO; AIFS/CW/TXOP) | ✅ full |
| **PCF** | ❌ missing (empty stub) |
| **HCCA** | ❌ placeholder module only ("INET currently does not implement HCCA") |
| **MCF** (mesh) | ❌ stub |
| **A-MPDU & A-MSDU** aggregation/deaggregation | ✅ present, **policy-gated** — default `Basic*` policies do *not* aggregate; must be enabled |
| **Block Ack** (originator + recipient; ADDBA/DELBA; immediate & delayed; compressed/basic/multi-TID; reordering buffer) | ✅ full |
| **Frame sequences** (RTS/CTS, data+ACK, BA exchanges, TXOP) | ✅ full — but **HT-specific TXOP sequences (`HtTxOpFs`) are a stub** (`selectHtTxOpSequence()` returns 0) |
| **Protection** (RTS/CTS, CTS-to-self, duration/NAV) | ✅ full |
| **Rate control** | ⚠️ ARF, AARF, Onoe only — **no Minstrel / Minstrel-HT** (the usual 11n/ac choice) |
| Fragmentation, lifetime, duplicate removal, per-TID sequence numbers | ✅ full |

There are essentially **no VHT-specific MAC behaviors** — 11ac reuses the same
EDCA / Block-Ack / aggregation machinery as 11n. Data-frame bandwidth and NSS are
set via `RateSelection` parameters (`dataFrameBandwidth`,
`dataFrameNumSpatialStreams`), **not negotiated**.

## 4. Management, frames, MIB

| Area | Status |
|------|--------|
| Infrastructure STA & AP, ad-hoc/IBSS, "simplified" variants | ✅ (`mgmt/`) |
| Scanning (active/passive), association/reassociation, beacons, probes, basic roaming/handover | ✅ (handover logic lives in the swappable `Ieee80211AgentSta`) |
| "Simplified" mgmt modules | ⚠️ static association only — no scan/auth/assoc, **cannot do handover** |
| Authentication | ⚠️ **open-system only** — `authType` omitted; no shared-key / EAP |
| **Security / encryption** (WEP, WPA/WPA2/WPA3, 802.11i/RSN, CCMP/GCMP, 802.11w MFP, 4-way handshake) | ❌ **none** — `protectedFrame` flag exists but is unused; RSN/capability IEs appear only as serializer comments |
| Information Elements in mgmt frames | ⚠️ **only SSID + Supported Rates.** **No HT Capabilities, HT Operation, VHT Capabilities, VHT Operation, EDCA Parameter Set, or QoS Capability IEs** |
| MIB (`mib/Ieee80211Mib.h`) | ✅ BSS/STA/AP membership, mode, `qos` flag — minimal |
| LLC (802.2/SNAP) & Portal (802.11 ↔ Ethernet bridge) | ✅ |

**Key structural consequence.** Because HT/VHT Capability/Operation elements are
*not* in the frame formats, STAs and APs **do not negotiate** 11n/ac parameters.
`opMode`, bandwidth, and NSS must be configured identically and by hand on every
node; the network never discovers or adapts them. This is the core reason 11ac is
labeled "incomplete," and it is the same negotiation gap the companion plan must
close (new capability IEs + MIB capability storage).

## 5. Examples, showcases, tests

- **11ac example:** `examples/wireless/lan80211ac/` — `opMode="ac"`,
  `numAntennas=8`, fixed `bitrate=693.3Mbps`, scalar radio. Default config uses
  **20 MHz only** (40/80/160 MHz are commented alternatives). Exercises 11ac in its
  simplest form — no multi-stream stress, no VHT-specific MAC combinations.
- **11n** appears in showcases (e.g. `showcases/wireless/txop` uses
  `n(mixed-2.4Ghz)`); aggregation/blockack showcases use `g`.
- **Tests:** generic MAC/PHY module + fingerprint tests exist; **no dedicated
  VHT/11ac test suite**.

---

## 6. User's Guide coverage (`doc/src/users-guide/ch-80211.rst`)

The chapter is **purely architectural** (Overview, MAC, PHY, Management, Agent).
It documents the modular MAC structure, the management variants (and that the
"Simplified" ones can't do handover), the agent's role in handover strategies,
and lists three radios (`Ieee80211ScalarRadio`, `Ieee80211DimensionalRadio`,
`Ieee80211UnitDiskRadio`).

It is most telling for what it **omits**: it never mentions WiFi generations,
`opMode`, MCS/HT/VHT, channel widths, or security. There is **no user-facing
documentation of 802.11ac/WiFi-5 specifics**, no statement of the n/ac "partial
support," and nothing on the absence of encryption — that acknowledgment exists
only in the `Ieee80211Mac.ned:33` comment. Any next-generation work should also
update this chapter.

---

## 7. Bottom line — what to reuse, what's missing at WiFi 5

**Solid / reusable foundation:**
- Full VHT PHY rate set: 20/40/80/160 MHz, MCS 0–9 (256-QAM), NSS 1–8, both GIs →
  realistic data rates and airtime.
- Mature, modular, shared MAC: EDCA QoS, A-MPDU/A-MSDU, full Block Ack, RTS/CTS,
  TXOP, replaceable policy modules.
- Both packet-level (fast) and bit-level (detailed) PHY abstractions with
  NIST/YANS error models.
- Clean PHY mode hierarchy and `opMode`/`ModeSet` selection mechanism — the exact
  extension points the companion plan builds on.

**Already missing at WiFi 5 (carry forward as constraints/known gaps):**
- **No real MIMO** — spatial streams are a rate multiplier only; **no MU-MIMO, no
  beamforming/sounding, no STBC**.
- **No LDPC** (TODO); convolutional codes only.
- **No 80+80 MHz**; 160 MHz present but unexercised in examples.
- **No HT/VHT capability negotiation** — IEs absent from frames; parameters must be
  hand-configured uniformly across nodes.
- **No security at all** (open networks only); **no PCF / HCCA**.
- **Rate control** lacks Minstrel-HT; supplied algorithms (ARF/AARF/Onoe) are
  legacy-oriented.
- **Nothing beyond WiFi 5 / 802.11ac** — later amendments are the subject of the
  companion plan.
