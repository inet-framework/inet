# WiFi 7 (802.11be / EHT) — Staged Development Plan

Staged roadmap for **delivering** WiFi 7 in INET, interleaving *fixing existing
issues* with *implementing the new EHT features*. Design rationale lives in the
companion [WiFi 7 design plan](wifi7-802.11be.md); this document is the *how/when*.

**Hard prerequisite:** the [WiFi 6 staged plan](wifi6-802.11ax-development-stages.md)
must be complete through its **Stage 3 (OFDMA core)** and ideally Stage 4. EHT
extends HE and reuses the Resource-Unit, multi-peer frame-sequence, Multi-STA
Block Ack, capability-negotiation, and MU machinery built there. This plan does
not re-list that foundation; it assumes it exists.

Guiding constraint (unchanged): existing scenarios keep working; new behavior is
behind new `opMode` values / capability flags.

---

## Stage 0 — EHT foundation fixes

**Goal:** close the few remaining model gaps EHT needs beyond the WiFi 6 work.

**Fix existing issues:**
- **Non-contiguous / wide bandwidth modeling** — 80+80 and 160+80 MHz not modeled
  in the VHT/HE bandwidth code; generalize the bandwidth representation to support
  these and 320 MHz.
- **STBC opt-in** — `mode/Ieee80211Vht/HtMode` hard-assume "STBC not used"; make
  it a capability flag (EHT extends STBC).
- **A-MPDU ceiling** — raise the aggregation limit to the EHT maximum (≈16 MB),
  building on the HE limit change from WiFi 6 Stage 2.

**Depends on:** WiFi 6 Stage 3+.
**Exit criteria:** bandwidth model expresses 320 MHz and punctured/non-contiguous
layouts; STBC toggleable.

---

## Stage 1 — EHT single-user PHY

**Goal:** an EHT link selectable via `opMode="be"`, no multi-RU/MLO yet.

**Implement (new):**
- `Ieee80211EhtMode` **sharing a parameterized core with `Ieee80211HeMode`**
  (EHT extends HE — favor sharing over a parallel class; see design plan).
- EHT MCS tables + **4096-QAM (4K-QAM)**.
- **320 MHz** channels; **up to 16 spatial streams**.
- EHT PPDU/preamble formats incl. EHT-SIG.
- Register the mode set; add `"be"` / `"be(320MHz)"` to the `opMode @enum`.

**Depends on:** Stage 0.
**Exit criteria:** unit tests: EHT SU rates & PPDU durations match the standard
(incl. 320 MHz, 4096-QAM, 16 SS); an example network runs an EHT SU link.

---

## Stage 2 — EHT capability negotiation & Multi-RU

**Goal:** EHT-capable discovery plus the first genuinely-new EHT MAC capability,
multi-RU / preamble puncturing — reusing the HE OFDMA machinery.

**Fix existing issues:**
- **Capability storage** — extend the per-peer capability record (added in WiFi 6
  Stage 2) with EHT fields in `mib/Ieee80211Mib.h`.
- **RU allocation generalization** — the HE RU model assumes one RU per STA;
  generalize to allow **multiple (incl. non-contiguous) RUs per STA**.

**Implement (new):**
- EHT Capabilities/Operation Information Elements + negotiation.
- **Multi-RU / preamble puncturing**: assign several RUs to one STA; tolerate
  punctured subchannels in scheduling and PPDU construction.
- Enhanced DL/UL OFDMA & MU-MIMO using the multi-RU allocator.

**Depends on:** Stage 1.
**Exit criteria:** two EHT nodes negotiate EHT capabilities; a scenario shows a
single STA served over multiple RUs and over a punctured channel.

---

## Stage 3 — Multi-Link Operation (MLO) — the headline feature

**Goal:** the defining WiFi 7 capability and the largest architectural item: one
logical link spanning **several radios/bands/channels at once**. This breaks the
one-radio-per-interface assumption underlying everything above, so it gets its own
detailed design doc before implementation.

**Implement (new):**
- **MLD layer** (Multi-Link Device) above multiple per-link radios + MACs: one
  association and one set of traffic flows mapped across several simultaneous
  links.
- **Link-set setup/teardown & negotiation**; per-link enable/disable.
- **Traffic-to-link steering**, including STR vs. non-STR operating modes
  (whether links can TX/RX simultaneously).
- **Cross-link sequence numbering / reordering** so frames split across links
  reassemble in order.

**Depends on:** Stage 2.
**Exit criteria:** an MLD associates over a 2-link set and sustains a single flow
across both links with correct in-order delivery; per-link enable/disable works.

---

## Stage 4 — R-TWT, hardening, validation

**Goal:** finish the feature set and make it trustworthy/demonstrable.

**Implement (new):** **Restricted TWT (R-TWT)** for latency-sensitive traffic.

**Validation & hardening:**
- Showcase/example networks for multi-RU, puncturing, and MLO.
- Regression/fingerprint tests for all WiFi ≤6 scenarios (no behavioral drift).
- Throughput/latency sanity-checks vs. published 802.11be expectations.

**Out of scope (v1):** Multi-AP coordination — a later 802.11be phase / future
amendment.

**Depends on:** Stages 1–3.
**Exit criteria:** green regression suite; documented MLO/multi-RU example results
within expected ranges of the standard.

---

### Stage dependency summary
`WiFi 6 (through Stage 3+) → EHT Stage 0 → 1 → 2 → 3 (MLO) → 4`. MLO is the
critical, highest-risk stage and warrants its own design doc before coding.
