# WiFi 6 Stage 3 — OFDMA: Detailed Development Plan

Detailed design+development plan for the **OFDMA core** of WiFi 6 — Stage 3 of the
[WiFi 6 staged plan](wifi6-802.11ax-development-stages.md), the centerpiece feature
from the [WiFi 6 design plan](wifi6-802.11ax.md). This document is detailed enough
to show the **new modules/classes**, where they attach to the current
architecture, and **how a user parameterizes** the feature.

**What OFDMA is here:** multiple STAs share one PPDU over disjoint **Resource
Units (RUs)**. Two directions: **DL-OFDMA** (AP schedules several STAs into one
downlink MU PPDU) and **UL-OFDMA** (AP sends a **Trigger frame**; solicited STAs
respond simultaneously, each on its assigned RU). Both require **Multi-STA Block
Ack** and a structural **single-peer → multi-peer** generalization of the MAC.

Class/file names below were taken from a scan of the current tree; treat them as
the intended attach points, not literal line numbers.

---

## 1. Prerequisites — required stages collected from the WiFi 5 & 6 plans

OFDMA sits on top of a chain of earlier work. These must be in place first.

### Must be complete before Stage 3 starts

| Prerequisite | Source plan / stage | Why OFDMA needs it |
|---|---|---|
| **HE single-user PHY** — `Ieee80211HeMode`, HE MCS, 1024-QAM, HE-SU PPDU, 4× numerology + GIs | WiFi 6 **Stage 1** | HE-MU / HE-TB PPDUs are composed from HE data modes; no SU mode → no MU mode. |
| **RU-ready bandwidth model** — bandwidth/GI data-driven, subcarrier count computable (not literal) | WiFi 6 **Stage 0** ⊃ WiFi 5 **A3** | An RU is a *subdivision of channel bandwidth*; the `Hz`-and-literal-subcarrier model must first become computable. |
| **Per-peer capability / operating-config state in MIB** + HE capability negotiation; rate selection honoring HE MCS sets | WiFi 6 **Stage 2** ⊃ WiFi 5 **B1** | The scheduler must know each STA's HE/OFDMA capability, supported RU sizes, MCS set, BA window. B1 is the keystone ("configuration over negotiation"). |
| **Block Ack scale-up** — 64→256 bitmap, drop data-frame fragmentation assumption; HE A-MPDU/A-MSDU limits | WiFi 6 **Stage 2** | Multi-STA BA and MU aggregation need the wider BA window and HE aggregation limits. |
| **TXOP & aggregation correctness** — `HtTxOpFs` stub fixed, VHT/HE A-MPDU bounds | WiFi 6 **Stage 0** ⊃ WiFi 5 **A2** | MU transmission is a TXOP construct; the multi-user TXOP sequence extends the (currently stubbed) HT/VHT TXOP path. |

### Should be complete / shared with later stages

| Prerequisite | Source | Relationship to OFDMA |
|---|---|---|
| **LDPC coded path** in error models | WiFi 6 Stage 0 ⊃ WiFi 5 **A1** | HE data mandates LDPC; needed for correct HE-MU PER, but the OFDMA *structure* can be built and tested with BCC first. |
| **Abstracted MIMO gain/SINR model** | WiFi 5 **§4** | Sets the modeling altitude OFDMA follows (see §2); strictly required only when OFDMA combines with **MU-MIMO** (WiFi 6 Stage 4 / WiFi 5 **B4**). |
| **HT/VHT-aware rate adaptation (Minstrel-HT)** | WiFi 5 **A4** | Per-RU/per-STA rate selection reuses it; a fixed RU/MCS map (§5) lets Stage 3 proceed without it. |

> Net: Stage 3 hard-depends on WiFi 6 **Stages 0–2** (which themselves absorb WiFi 5
> **A2, A3, B1**). WiFi 5 **A1, §4, A4** are soft dependencies it can stub around.

## 2. The one real architectural decision — how an MU PPDU is modeled

Today the PHY is strictly **one transmission = one mode = one duration = one
receiver**: `Ieee80211Transmission` carries a single `const IIeee80211Mode *mode`,
the error model derives one PER, and one receiver decodes the whole PPDU
(`Ieee80211Transmitter::createTransmission`, `Ieee80211ErrorModelBase`,
`Ieee80211Receiver`). OFDMA breaks this — one PPDU, many users, disjoint RUs.

**Recommended approach (matches the WiFi 5 §4 abstraction altitude):** keep it as
**one `Ieee80211Transmission` with one airtime/duration**, but let it carry a
**list of per-RU sub-allocations** `{RU, target STA, HE MCS/mode, tx power}`. The
error model computes **per-RU PER** from each RU's SINR and mode; each addressed
STA decodes **only its own RU**. Per-RU SINR comes from the existing pathloss/SINR
machinery plus an abstracted power split, with an **optional cross-RU leakage
penalty** (the frequency-domain analogue of §4's MU-MIMO cross-talk penalty),
exposed as a scenario parameter.

**Rejected for v1:** a true subcarrier/precoder physical model — far below INET's
altitude and not needed for the MAC-level studies (airtime, scheduling, aggregate
throughput) OFDMA serves. The per-RU SINR adjustment is **factored behind one
hook** so a future detailed model can replace it without touching the MAC.

## 3. New & changed modules/classes

Grouped by subsystem. "New" = create; "Change" = generalize existing, keeping the
single-user path intact and behavior-identical when OFDMA is off.

### 3a. PHY — Resource Units, HE-MU/HE-TB PPDU, multi-destination transmission
*(dir: `physicallayer/wireless/ieee80211/mode/` and `.../packetlevel/`)*

- **New `Ieee80211HeRu`** (mode/) — an RU descriptor: size enum
  (26/52/106/242/484/996/2×996-tone), index/offset within the channel, and its
  subcarrier count. Becomes the unit the (now computable) subcarrier count maps to.
- **New `Ieee80211HeMuPpduMode`** (mode/) — an HE-MU PPDU built from a list of
  `{Ieee80211HeRu, Ieee80211Hemcs, NSS, target STA}`; models HE-SIG-B per-user
  fields. **New `Ieee80211HeTbPpduMode`** — the UL trigger-based counterpart.
  Both reuse the per-STA HE data modes from Stage 1.
- **Change `Ieee80211Transmission`** — add `std::vector<RuAllocation> ruAllocations`
  (RU, `MacAddress`, mode, power); `getMode()` stays for SU, plus
  `getModeFor(receiverAddress)` for MU. Single-user transmissions leave the vector
  empty → unchanged.
- **Change `Ieee80211ErrorModelBase` / `Ieee80211NistErrorModel` / `…Yans`** —
  when `ruAllocations` is non-empty, compute PER **per RU** (smaller RU = fewer
  subcarriers → different curve) for the addressed receiver; add the optional
  cross-RU leakage term (§2).
- **Change `Ieee80211Transmitter` / `Ieee80211Receiver`** — transmitter reads a new
  **`Ieee80211MuAllocationReq`** tag (below) to populate `ruAllocations`; receiver
  selects its RU via `getModeFor(localAddress)`.

### 3b. MAC — new frame types
*(dir: `linklayer/ieee80211/mac/` + `…/mgmt/Ieee80211MgmtFrame.msg` + serializers)*

- **New `Ieee80211TriggerFrame`** — control frame carrying common info + a per-user
  list `{AID, RU, MCS, target RSSI}`; serializer + dissector entries.
- **New `Ieee80211MultiStaBlockAck`** (or extend `Ieee80211BlockAck`) — per-STA BA
  info blocks in one frame.
- **New `Ieee80211MuRts` / MU-BAR** *(optional this stage)* — MU protection / BA
  solicitation.

### 3c. MAC — multi-user coordination (the heart)
*(dir: `linklayer/ieee80211/mac/coordinationfunction/`, `.../framesequence/`, new `.../ofdma/`)*

- **New `IOfdmaScheduler`** (contract/) **+ `Ieee80211HeScheduler`** (new `mac/ofdma/`) — at an
  OFDMA-capable AP, when an `Edcaf` wins a TXOP (`Hcf::channelGranted`), decides
  whether to transmit MU and produces the **RU allocation map** (which STAs → which
  RUs → which MCS), pulling candidate frames from multiple `Edcaf`s'
  `InProgressFrames`. Referenced as a submodule from `Hcf` (don't rewrite `Hcf`).
  **Full C++ contract in §3c.1.**
- **New frame sequences** (`framesequence/`, implementing `IFrameSequence`):
  - **`DlOfdmaFs`** — one `MuTransmitStep` (the MU PPDU to N STAs) followed by a
    `MuReceiveStep` collecting N Block Acks (immediate or MU-BAR-solicited).
  - **`UlOfdmaFs` / `TriggerFrameFs`** — a `TransmitStep` (Trigger frame), a
    `MuReceiveStep` collecting N simultaneous HE-TB responses, then a
    `TransmitStep` (Multi-STA BlockAck).
  - Both wired into **`HcfFs::selectHcfSequence`** / **`TxOpFs`** as new
    alternatives, gated by "AP is OFDMA-enabled and the scheduler chose MU";
    otherwise the existing SU sequence runs unchanged.
- **New step types** (`framesequence/FrameSequenceStep.h`): **`MuTransmitStep`**
  (returns the composed MU PPDU packet + its `Ieee80211MuAllocationReq` tag) and
  **`MuReceiveStep`** (`std::vector` of responses keyed by STA/RU within one
  timeout window) — the multi-user analogues of `ITransmitStep`/`IReceiveStep`.
- **Change `FrameSequenceContext`** — add an optional **`MultiUserContext`**: a
  vector of per-STA sub-contexts (per-STA `InProgressFrames`, TID, BA agreement,
  RU/mode). This is the single-peer→multi-peer generalization. The existing single
  `MacAddress` / `QoSContext` path stays for SU; MU context is opt-in.
- **Change `InProgressFrames`** — add per-STA frame selection
  (`getFramesToTransmit(selection)`) so the scheduler can compose subframes for
  several STAs; the existing `getFrameToTransmit()` is unchanged.

### 3c.1 The scheduler's C++ interface (`IOfdmaScheduler`)

The scheduler is the one genuinely new control-plane component, so its contract is
spelled out here. It follows the established INET MAC pattern: a pure-abstract
`INET_API` interface in `mac/contract/` (paired with a NED `moduleinterface` of the
same name), and a concrete `SimpleModule` implementation referenced by `Hcf` as a
`like` submodule — exactly how `IRateControl`/`rateControl` and
`QosRateSelection`/`rateSelection` are wired today. The interface header lives in
`mac/contract/`; the implementation and the value types below live in `mac/ofdma/`.

**Single call site.** `Hcf::channelGranted` is the only place the scheduler runs.
Today it does `startTxop(ac)` then `startFrameSequence(ac)`; OFDMA inserts exactly
one call between them:

```cpp
void Hcf::channelGranted(IChannelAccess *channelAccess) {
    auto edcaf = check_and_cast<Edcaf *>(channelAccess);
    AccessCategory ac = edcaf->getAccessCategory();
    edcaf->getTxopProcedure()->startTxop(ac);
    ... // internal-collision handling unchanged
    muSchedule = scheduler ? scheduler->scheduleTxop(edcaf, edca) : MuSchedule::none();
    startFrameSequence(ac);   // buildContext() folds muSchedule into the context
}
```

`scheduler == nullptr` (the default — `ofdmaEnabled = false` ⇒ no submodule) or a
schedule whose `isMu()` is false ⇒ the existing single-user path runs byte-for-byte
as today. This is the only edit to `channelGranted`.

**The data the scheduler returns** — two small header-only value types in
`mac/ofdma/`:

```cpp
// One per-user resource-unit assignment within an MU PPDU or a Trigger frame.
class INET_API RuAllocation {
  public:
    Ieee80211HeRu ru;                                     // size enum + index/offset in the channel
    MacAddress staAddress;                                // DL: target STA;  UL: solicited STA
    Tid tid = -1;                                         // TID this RU serves / solicits
    const physicallayer::IIeee80211Mode *mode = nullptr;  // per-RU HE data mode (MCS/NSS); borrowed
    W txPower = W(0);                                     // DL: per-RU power split (§2)
    W targetRssi = W(0);                                 // UL: target receive power placed in the Trigger
};

class INET_API MuSchedule {
  public:
    enum class Direction { NONE, DOWNLINK, UPLINK };
    Direction direction = Direction::NONE;
    std::vector<RuAllocation> allocations;                // empty ⇒ fall back to SU
    bool isMu() const { return direction != Direction::NONE && !allocations.empty(); }
    static MuSchedule none() { return {}; }
};
```

`RuAllocation` is **the same element type** the PHY-side `Ieee80211MuAllocationReq`
tag (§3a/§3e) carries: the schedule produced here is exactly what `MuTransmitStep`
later attaches to the composed PPDU, so there is a single allocation representation
flowing MAC → PHY (no translation layer).

**The interface itself** — one decision method plus a capability query:

```cpp
// mac/contract/IOfdmaScheduler.h   (+ NED: moduleinterface IOfdmaScheduler)
class INET_API IOfdmaScheduler {
  public:
    virtual ~IOfdmaScheduler() {}

    // Called once per won TXOP, from Hcf::channelGranted, after startTxop().
    //  - `winner` is the Edcaf that won contention (its AC seeds the schedule);
    //  - `edca` gives read access to every AC's InProgressFrames so the scheduler
    //    can co-schedule frames for several STAs into one PPDU.
    // Returns a DL or UL schedule, or MuSchedule::none() to keep the SU path.
    virtual MuSchedule scheduleTxop(Edcaf *winner, Edca *edca) = 0;

    // True if this AP may transmit MU at all: ofdmaEnabled && (dlOfdma || ulOfdma)
    // && at least one OFDMA-capable associated peer. HcfFs may also gate on this.
    virtual bool isOfdmaEnabled() const = 0;
};
```

That is the whole contract. Everything else — RU sizing, user grouping, the
fixed-vs-dynamic policy — is internal to `Ieee80211HeScheduler`, keeping
`Hcf`/`HcfFs` ignorant of scheduling policy, the same way `Hcf` is ignorant of
rate-control policy today.

**What `scheduleTxop` reads, per direction:**
- **DL:** walks `edca->getEdcaf(ac)->getInProgressFrames()` across ACs, groups the
  outstanding frames by destination `MacAddress` (only OFDMA-capable peers, per the
  Stage-2 MIB), and assigns up to `maxUsersPerMuPpdu` of them to RUs. It only
  **inspects** (`InProgressFrames::getOutstandingFrames()`); it does **not** dequeue
  — `MuTransmitStep` later pulls the actual subframes for the chosen STAs via the
  new per-STA `getFramesToTransmit(selection)` (§3c).
- **UL:** no local data to pick — it selects which associated STAs to solicit
  (round-robin / buffer-status-report driven) and fills each `RuAllocation` with the
  RU, the MCS the STA must use, and `targetRssi`. `UlOfdmaFs` turns the returned
  allocations into the `Ieee80211TriggerFrame` per-user list.

**Per-RU MCS is not decided in the scheduler directly.** For
`ruAllocationPolicy = "dynamic"` it calls the new `Ieee80211OfdmaRateSelection`
(§3e) to fill `RuAllocation::mode` from per-STA history keyed by `(MacAddress, RU)`;
for `"fixed"` it reads the parsed `fixedRuMap`. This keeps rate logic in the
rate-selection module, mirroring how `Hcf::transmitFrame` defers to
`rateSelection->computeMode(...)`.

**How the schedule reaches the frame sequence.** `Hcf::buildContext` stores the
`MuSchedule` in the new `MultiUserContext` on `FrameSequenceContext` (§3c).
`HcfFs::selectHcfSequence` — already an `AlternativesFs` selector returning an
alternative index — then branches on it, leaving the SU branch untouched:

```cpp
int HcfFs::selectHcfSequence(AlternativesFs *fs, FrameSequenceContext *context) {
    if (auto mu = context->getMultiUserContext()) {
        if (mu->getSchedule().direction == MuSchedule::Direction::DOWNLINK) return DL_OFDMA_ALT;
        if (mu->getSchedule().direction == MuSchedule::Direction::UPLINK)   return UL_OFDMA_ALT;
    }
    return SU_ALT;   // existing single-user selection, unchanged
}
```

So the scheduler decides **once** at TXOP grant; the frame sequence and its
`MuTransmitStep`/`MuReceiveStep` are pure mechanism consuming the immutable
`MuSchedule` from the context. No scheduler call happens mid-sequence.

**NED wiring** (the submodule the §5 params configure):

```ned
// in module Hcf
scheduler: <default("")> like IOfdmaScheduler if typename != "" {
    parameters: @display("p=350,300");
}
```

No `typename` ⇒ no submodule ⇒ `Hcf::scheduler == nullptr` ⇒ SU. Setting
`scheduler.typename = "Ieee80211HeScheduler"` (gated behind `ofdmaEnabled`)
activates it; its `dlOfdma` / `ulOfdma` / `maxUsersPerMuPpdu` / `ruAllocationPolicy`
/ `fixedRuMap` params (§5) are read in `initialize()`. `Hcf` resolves the pointer
once via `check_and_cast<IOfdmaScheduler *>(getSubmodule("scheduler"))`, exactly as
it resolves `rateSelection`/`rateControl`.

### 3d. MAC — Multi-STA Block Ack
*(dir: `linklayer/ieee80211/mac/blockack/`, `.../recipient/`)*

- **New `RecipientMultiStaBlockAckProcedure`** (or extend
  `RecipientBlockAckProcedure`) — on receiving a Trigger/MU-BAR, build this STA's
  per-STA BA info block.
- **Change `Originator…BlockAckAgreement` lookup** — parse a `Multi-StaBlockAck`
  addressed to several STAs and route each BA block to its `(addr, TID)` agreement.
  Agreements stay keyed `(addr, TID)`; only the *frame* becomes multi-STA.
- Depends on the **256-entry BA window** (`QosAckHandler`) from Stage 2.

### 3e. MAC — per-RU rate/mode selection & the carrier tag
*(dir: `linklayer/ieee80211/mac/rateselection/`, PHY tag defs)*

- **New `Ieee80211OfdmaRateSelection`** (extends `QosRateSelection`) — for MU
  frames returns a **per-STA/per-RU** mode instead of one mode; reuses the existing
  `lastTransmittedFrameMode` map keyed by `MacAddress`, extended with RU.
- **New `Ieee80211MuAllocationReq` tag** — carries `std::vector<RuAllocation>`;
  replaces the single `Ieee80211ModeReq` tag for MU PPDUs (`Hcf::setFrameMode`
  attaches `ModeReq` for SU; the scheduler attaches `MuAllocationReq` for MU).

## 4. How it fits the existing data flow

The existing path (`Hcf::processUpperFrame` → `Edca`/`Edcaf` backoff →
`Hcf::channelGranted` → `FrameSequenceHandler::startFrameSequence(HcfFs)` → steps →
`Tx` → radio) is preserved. OFDMA inserts at three points:

**DL-OFDMA**
```
Edcaf wins TXOP → Hcf::channelGranted
  → Ieee80211HeScheduler: MU? build RU map (STAs→RUs→MCS)
  → HcfFs selects DlOfdmaFs
      MuTransmitStep: compose MU PPDU (subframes + Ieee80211MuAllocationReq tag) → Tx → radio
      [each STA decodes only its RU via getModeFor(localAddr); per-RU PER]
      MuReceiveStep: collect N BlockAcks (one timeout window)
  → frame sequence finished
```

**UL-OFDMA**
```
Scheduler builds Ieee80211TriggerFrame (per-user RU/MCS)
  → HcfFs selects UlOfdmaFs / TriggerFrameFs
      TransmitStep: Trigger frame → Tx → radio
      MuReceiveStep: N STAs send HE-TB PPDUs simultaneously on their RUs; collect by STA/RU
      TransmitStep: Ieee80211MultiStaBlockAck
```

Single-peer assumptions replaced (all opt-in, SU path untouched):
`FrameSequenceContext` (→ `MultiUserContext`), one-receiver `Ieee80211Transmission`
(→ `ruAllocations`), one-PER error model (→ per-RU), one-RA Block Ack (→ multi-STA).

## 5. User parameterization

Built on the **existing `opMode` selection mechanism** plus a small set of new,
**default-off** parameters, following the WiFi 5 **B1** principle of *direct
control over the outcome* (a fixed RU schedule for reproducible studies, a dynamic
scheduler for realistic ones).

**Select the generation (existing mechanism, new enum value):**
```ini
*.host*.wlan[0].opMode = "ax"          # propagates to radio + mac.modeSet (Ieee80211Interface.ned)
```

**Enable OFDMA at the AP (new params on the HCF / scheduler):**
```ini
*.ap.wlan[0].mac.hcf.ofdmaEnabled        = true                  # default false → HE single-user
*.ap.wlan[0].mac.hcf.scheduler.typename  = "Ieee80211HeScheduler"
*.ap.wlan[0].mac.hcf.scheduler.dlOfdma   = true
*.ap.wlan[0].mac.hcf.scheduler.ulOfdma   = true
*.ap.wlan[0].mac.hcf.scheduler.maxUsersPerMuPpdu  = 8
*.ap.wlan[0].mac.hcf.scheduler.ruAllocationPolicy = "dynamic"    # @enum("fixed","dynamic")
# "fixed" → direct, reproducible RU map (WiFi 5 B1 "configuration over negotiation"):
*.ap.wlan[0].mac.hcf.scheduler.fixedRuMap = "sta1:RU242:mcs7, sta2:RU242:mcs5, ..."
```

**PHY/abstraction knobs (scenario calibration, per §2):**
```ini
*.*.wlan[0].radio.receiver.errorModel.crossRuLeakage = 0dB       # optional MU penalty
```

**Capability state (from the B1 / Stage 2 operating-config model, set directly):**
```ini
*.host*.wlan[0].mac.mib.heCapabilities.ofdmaSupported = true
*.host*.wlan[0].mac.mib.heCapabilities.maxRus         = 4
```

**STA side:** normally just `opMode="ax"` + capabilities — UL-OFDMA participation
is *triggered*, needing no manual config; an opt-out flag is available.

**Backward compatibility:** `ofdmaEnabled = false` (default) → an `"ax"` network
behaves exactly as HE single-user (Stage 1/2); every WiFi ≤5 scenario is untouched.

## 7. Sub-sequencing & testing (Stage 3 internal)

Test-driven per the WiFi 5 governing principle — each item starts with its test.

1. **3.0 Foundations (highest risk, do first):** the PHY multi-destination
   transmission + per-RU error model, and the `FrameSequenceContext` multi-peer
   generalization — both with the single-user path proven unchanged (regression).
2. **3.1 DL-OFDMA:** AP-controlled, simplest; scheduler + `DlOfdmaFs` + per-RU
   tags + N-BlockAck collection.
3. **3.2 UL-OFDMA + Trigger frames:** `Ieee80211TriggerFrame`, `UlOfdmaFs`,
   synchronized HE-TB reception.
4. **3.3 Multi-STA Block Ack:** the multi-STA BA frame + recipient/originator
   handling tying 3.1/3.2 together.

**Tests:** module tests for RU allocation, trigger round-trip, multi-STA BA
bitmap, per-RU PER; full-sim for N-STA DL throughput scaling and UL-OFDMA
efficiency vs. EDCA contention; backward-compat regression on all SU scenarios.

**Exit criteria:** an AP serves N STAs in one DL-OFDMA PPDU and solicits a
synchronized UL-OFDMA response acknowledged by a single Multi-STA BA; aggregate
throughput scales with users; all WiFi ≤5 and HE-SU scenarios unchanged.
