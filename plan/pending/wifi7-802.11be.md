# Implementing IEEE 802.11 WiFi 7 (802.11be / EHT) in INET

High-level plan for adding the **new** WiFi 7 capabilities. Intentionally not a
detailed design — it sets the altitude, scope, and sequencing.

**Prerequisite:** this plan builds directly on the
[WiFi 6 plan](wifi6-802.11ax.md). WiFi 7 (EHT) extends the HE PHY and reuses the
Resource-Unit and multi-user frame-sequence machinery introduced for WiFi 6.
Do that work first; this plan assumes it exists.

WiFi 7 = amendment **802.11be** = **EHT** (Extremely High Throughput) PHY family.

---

## 1. Scope — what's new to build (on top of WiFi 6)

- **PHY**
  - New `Ieee80211EhtMode` (same shape as the HE mode), EHT MCS tables,
    **4096-QAM (4K-QAM)**.
  - **320 MHz** channels (6 GHz).
  - Up to **16 spatial streams**.
  - EHT PPDU/preamble formats (EHT-SIG).
- **MAC**
  - **Multi-Link Operation (MLO)** — *the headline feature; see §4.*
  - **Multi-RU / preamble puncturing** — assign multiple (incl. non-contiguous)
    RUs to one STA; tolerate punctured subchannels.
  - Enhanced DL/UL OFDMA & MU-MIMO reusing the WiFi 6 machinery.
- **Mgmt/MIB**
  - EHT capability Information Elements; MLO link-set negotiation.
  - *(Out of scope for v1: Multi-AP coordination — a later 802.11be phase.)*

---

## 2. Extension strategy — let the user pick the WiFi version

Same guiding principles as the [WiFi 6 plan](wifi6-802.11ax.md): **composability
over versioning** (enough degrees of freedom to combine components into WiFi 5/6/7
— each optionally backward compatible — and even non-standard configurations),
**parameterize, don't proliferate** (prefer one parameterized module over a
per-generation module unless it would overcomplicate things), and **keep the
existing `opMode` selection mechanism**.

1. **EHT PHY mode — extend/parameterize, don't clone.** Realize EHT rates by
   adding EHT parameters (320 MHz, 4096-QAM, up to 16 spatial streams) to the
   shared mode / data-mode / MCS structure; add dedicated EHT classes only where
   the differences (e.g. EHT PPDU / EHT-SIG formats) warrant it. EHT extends HE,
   so favor sharing a parameterized core with the HE mode rather than a parallel
   class.
2. **Extend `Ieee80211ModeSet` + the `opMode` enum** with values — e.g. `"be"`,
   `"be(320MHz)"`.
3. **Reuse the WiFi 6 multi-user machinery** — RUs, trigger-based frame sequences,
   multi-STA Block Ack — extending it for multi-RU/puncturing rather than building
   anew.
4. **MLO is a new architectural layer**, not just another mode — see §4. It is the
   one piece that does not fit the single-radio mode pattern and needs its own
   structure spanning multiple links/radios.
5. **Layered, optional rollout.** Single-user EHT (320 MHz, 4096-QAM, 16 SS) is
   usable without multi-RU or MLO; those are separable increments on top.

Net effect for the user: `*.opMode = "be"` selects the generation; booleans /
link-set config enable multi-RU and MLO when desired.

---

## 3. WiFi 7 feature categorization (802.11be / EHT)

Builds on the WiFi 6 foundation; implemented in roughly this order.

- **Multi-Link Operation (MLO)** — *the headline feature; see §4.*
- **320 MHz channels** (6 GHz).
- **4096-QAM (4K-QAM)** + new EHT MCS tables.
- **Multi-RU / preamble puncturing** — multiple (incl. non-contiguous) RUs per STA.
- **Up to 16 spatial streams**; enhanced DL/UL OFDMA & MU-MIMO.
- **EHT PPDU formats / EHT-SIG.**
- **Restricted TWT (R-TWT)** for latency-sensitive traffic.

### Suggested delivery order
1. EHT single-user PHY (320 MHz, 4096-QAM, 16 SS) + `opMode="be"`.
2. Multi-RU / preamble puncturing; enhanced MU.
3. **Multi-Link Operation (MLO)**.

---

## 4. Multi-Link Operation (MLO) — the central feature

MLO is the defining capability of WiFi 7 and the largest architectural item,
because a single logical link now spans **several radios/bands/channels at once**
— this breaks the one-radio-per-interface assumption that everything else
(including the WiFi 6 work) is built on.

- **Architecture:** a logical MLD (Multi-Link Device) sits above multiple
  per-link radios + MACs; needs a layer that maps one association and one set of
  traffic flows across several simultaneous links.
- **Link management:** setup/teardown of the link set, per-link enable/disable,
  and steering traffic across links (and the modes — STR vs. non-STR — that govern
  whether links can transmit/receive simultaneously).
- **Sequence numbering / reordering across links** so frames split over links are
  reassembled in order.
- **Capability negotiation** of the link set during association.

Because MLO touches association, addressing, queuing and scheduling across
multiple interfaces, it is sequenced last and warrants its own detailed design
doc before implementation.

---

## 5. Cross-cutting work & validation (brief)
- **Capability negotiation:** EHT Information Elements + MLO link-set negotiation
  in beacons/assoc frames, with MIB capability storage.
- **Opt-in by design:** new behavior strictly behind the new `opMode` values /
  capability flags; multi-RU and MLO individually toggleable.
- **Validation:** unit tests per new EHT PHY mode (rates/durations vs. the
  standard) and new example/showcase networks demonstrating multi-RU and MLO.
  Sanity-check throughput/latency against published 802.11be expectations.
