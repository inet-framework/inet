# Implementing IEEE 802.11 WiFi 6 (802.11ax / HE) in INET

High-level plan for adding the **new** WiFi 6 capabilities. Intentionally not a
detailed design — it sets the altitude, scope, and sequencing. Each phase gets
its own detailed design doc later.

This is the **foundation plan**: it introduces the HE PHY and the multi-user
(OFDMA) machinery that WiFi 7 then builds on. See the companion
[WiFi 7 plan](wifi7-802.11be.md), which depends on this one.

WiFi 6/6E = amendment **802.11ax** = **HE** (High Efficiency) PHY family.

---

## 1. Scope — what's new to build

The work, grouped by where it lands:

- **PHY**
  - New `Ieee80211HeMode` (preamble/signal/data submodes, like the existing mode
    classes), HE MCS tables, **1024-QAM**.
  - HE PPDU/preamble formats — HE-SU, HE-MU, HE-TB, HE-ER-SU (new HE-SIG).
  - **Resource Units (RUs)** as the new bandwidth abstraction (for OFDMA).
  - New OFDM numerology — 4× longer symbol, guard intervals 0.8/1.6/3.2 µs.
  - 6 GHz band channelization (WiFi 6E).
- **MAC** (multi-user — none of this exists today)
  - OFDMA scheduling (DL & UL), trigger frames, multi-STA Block Ack.
  - DL/UL MU-MIMO, BSS coloring / spatial reuse, Target Wake Time (TWT).
- **Mgmt/MIB**
  - HE capability Information Elements + capability negotiation so features can
    be advertised and toggled.

## 2. Extension strategy — let the user pick the WiFi version

Guiding design principles:

- **Composability over versioning.** Design the WiFi components with enough
  degrees of freedom (parameters, pluggable sub-behaviors) that *combining* them
  can express WiFi 5, WiFi 6, and later generations — each *optionally* backward
  compatible — and even non-standard / non-compliant configurations. A "WiFi
  version" should be a particular combination of parameters, not a hard-wired
  variant.
- **Parameterize, don't proliferate.** Prefer a single module/class driven by
  parameters over a separate module per WiFi generation, wherever it fits cleanly
  without overcomplicating the module. Split into a dedicated module/class only
  when the differences are large enough that one parameterized module would
  become unwieldy.
  *Note:* the standard's distinct names (HT/VHT/HE/EHT) are a hint, not proof, of
  structural difference — the existing `Ieee80211HtMode` / `Ieee80211VhtMode` were
  split on the assumption they were more different than similar, which in
  hindsight is debatable. Decide per component on actual overlap: the single-user
  rate / data-mode / MCS machinery is largely shareable across generations (mostly
  different tables/parameters), whereas genuinely new structure — OFDMA RU framing,
  MU PPDU and preamble/SIG formats — earns its own components. HE↔EHT in particular
  are close (EHT extends HE), so favor a shared parameterized core there. Leave the
  existing HT/VHT classes as-is unless unification falls out naturally.
- **Keep INET's existing selection mechanism.** A user picks a generation by
  setting one parameter (`opMode`) on the radio; PHY rates, timings, and
  capabilities follow from it. WiFi 6 slots into the same pattern. (This mechanism
  is reused by the WiFi 7 plan.)

1. **HE PHY mode — extend/parameterize the existing structure where it fits.**
   Realize HE rates by adding HE parameters (1024-QAM, RU-based bandwidths, new
   guard intervals / numerology) to the existing mode / data-mode / MCS structure
   rather than cloning a parallel class hierarchy; introduce dedicated HE classes
   only for the parts that genuinely differ (e.g. HE PPDU/preamble formats). Reuse
   legacy preamble components for backward compatibility.

2. **Extend `Ieee80211ModeSet` + the `opMode` enum.** Register the new mode set
   and add enum values — e.g. `"ax"`, `"ax(6Ghz)"` — to the `opMode @enum(...)` in
   `Ieee80211Radio.ned` / `Ieee80211Transmitter.ned` / `Ieee80211Receiver.ned`.
   Selecting the version stays a one-line config change.

3. **Gate MAC behavior behind capability flags, default off.** OFDMA / MU
   fundamentally change channel access. Capability flags (in the MIB, negotiated
   via new HE IEs or set directly) turn each multi-user feature on, so default
   behavior is unchanged unless explicitly enabled.

4. **Subclass rather than rewrite the MAC.** Add an HE-aware coordination path
   (extension of `Hcf`) and new **frame-sequence** variants for the multi-user
   exchanges, leaving the existing channel-access primitives intact. Frame
   sequences are the natural plug-in point.

5. **Layered, optional rollout.** Single-user HE (new PHY rates, formats,
   1024-QAM, bandwidths) must be usable *without* OFDMA. OFDMA and MU-MIMO are
   added as separable increments on top, so each step is testable.

Net effect for the user: `*.opMode = "ax"` selects the generation; additional
booleans enable OFDMA / MU when desired.

## 3. WiFi 6 feature categorization (802.11ax / HE)

Implemented in roughly this priority order; **OFDMA first** (see §4).

- **OFDMA (DL & UL) with Resource Units** — *the headline feature; see §4.*
- **Trigger frames** — AP solicits/schedules synchronized uplink MU transmissions.
- **DL/UL MU-MIMO** — up to 8 simultaneous users (spatial multiplexing).
- **Multi-STA Block Ack** — one BA frame acknowledges many STAs.
- **1024-QAM** + new HE MCS tables.
- **HE PPDU formats / preamble** — HE-SU, HE-MU, HE-TB, HE-ER-SU (new HE-SIG).
- **New OFDM numerology** — 4× longer symbol, GIs of 0.8/1.6/3.2 µs.
- **Spatial reuse / BSS Coloring** — color-aware deferral to raise channel reuse.
- **Target Wake Time (TWT)** — scheduled wake for efficiency/power (lower priority).
- **6 GHz band operation (WiFi 6E)** — new channelization.

### Suggested delivery order
1. HE single-user PHY (modes, MCS, formats, bandwidths) + `opMode="ax"`.
2. **DL OFDMA**, then **UL OFDMA + trigger frames**, then multi-STA Block Ack.
3. DL/UL MU-MIMO; BSS coloring; TWT.

---

## 4. OFDMA — the central feature

OFDMA is the defining capability of WiFi 6 and the hardest part, because it turns
a single contended channel into many **Resource Units (RUs)** carrying distinct
per-STA transmissions inside one PPDU. It touches PHY *and* MAC.

**PHY side:**
- Model bandwidth as a set of **Resource Units** (e.g. 26/52/106/242/484/996-tone)
  rather than one monolithic channel, so several STAs share one transmission.
- HE data modes parameterized per-RU (each RU has its own MCS / spatial streams);
  HE-MU/HE-TB PPDU structure.
- Error/duration computation must work per-RU within a shared PPDU.

**MAC side (the bulk of the work):**
- **DL OFDMA:** the AP schedules multiple STAs' frames into one MU PPDU — needs a
  scheduler/resource allocator deciding which STA gets which RU.
- **UL OFDMA:** AP sends a **Trigger frame**; solicited STAs respond
  *simultaneously* on assigned RUs — needs new trigger-based frame-sequence
  variants and synchronized-response handling.
- **Multi-STA Block Ack** to acknowledge the many concurrent users.
- Extend the **frame-sequence context** to track multiple STAs / RUs / per-STA
  rates within a single exchange (today it assumes one peer).

**Why it's sequenced first among MAC features:** trigger frames, multi-STA BA,
MU-MIMO (and, later, WiFi 7's multi-RU and MLO) all reuse the RU abstraction and
the multi-user frame-sequence machinery introduced here. Get OFDMA right and the
rest compose on top of it.

## 5. Cross-cutting work & validation (brief)
- **Capability negotiation:** new HE Information Elements in beacons/assoc frames +
  MIB capability storage, so features are advertised and toggled cleanly.
- **Opt-in by design:** new behavior strictly behind the new `opMode` values /
  capability flags.
- **Validation:** unit tests per new HE PHY mode (rates/durations vs. the
  standard) and new example/showcase networks demonstrating OFDMA and MU-MIMO.
  Sanity-check throughput/latency against published 802.11ax expectations.
