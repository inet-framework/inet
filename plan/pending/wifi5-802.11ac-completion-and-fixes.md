# Completing & Fixing IEEE 802.11 WiFi 5 (802.11ac / VHT) in INET

Staged, **test-driven** development plan to bring INET's existing 802.11ac / VHT
support from "partial/incomplete" (its own description, `Ieee80211Mac.ned:33`) up
to a complete, validated WiFi 5 implementation — **and nothing beyond it**.

Forward-looking counterpart to the baseline study in
[`wifi5-802.11ac-current-model.md`](wifi5-802.11ac-current-model.md). High-level
by design: it sets scope, sequencing, and risk. Each item gets its own detailed
design doc — starting with its tests — before implementation.

The work is split into two clearly separated tracks:

- **Part A — Completing existing, incomplete features.** Finishing or fixing
  things that are already partly in the tree (stubs, TODOs, untested paths).
- **Part B — New WiFi 5 features.** Capabilities that are entirely absent today
  (operating-configuration model, beamforming, DL MU-MIMO, optional negotiation).

**Scope: WiFi 5 / 802.11ac (VHT) only.** Explicitly *out of scope*:
- 802.11ax / WiFi 6, 802.11be / WiFi 7 and anything later — separate plan.
- Link-layer security (WEP/WPA/WPA2/WPA3, RSN, 802.11w): genuinely missing, but a
  cross-cutting 802.11 concern, not a VHT feature. Track separately.

---

## 1. Governing principles

1. **Test-driven — tests come first.** Every fix or feature *starts* by writing
   its test, which then drives the implementation to green. A test is either:
   - a **full simulation** (an example/showcase network turned into a
     fingerprint/regression test, asserting throughput/latency/airtime, or exact
     rate/duration), or
   - a **module test with mockups** (drive a single class/submodule — a frame
     sequence, the rate-control algorithm, an error model — with a scripted/mock
     context and assert its decisions directly).

   Validation is therefore *not* a final stage; it is the entry point of each
   item below. The only thing deferred to the end is documentation.

2. **Configuration over negotiation — model the *result*, not the *process*.**
   Simulating the full capability-negotiation handshake is rarely important in
   itself: in most studies the statistical results depend only on the *outcome*
   of negotiation (the operating configuration actually in use), not on the
   exchange that produced it. And users very often want **direct control** over
   that outcome — i.e. to state exactly which configuration is simulated — where a
   negotiation handshake would just be in the way. Therefore:
   - The first-class deliverable is an **operating-configuration / capability
     state** (bandwidth, NSS, GI, MCS set, coding, beamforming/MU flags) that is
     **directly settable by the user** and consumed by PHY+MAC. This is the
     default path (Part B1).
   - Actually *simulating* the negotiation that derives this state is an
     **optional, opt-in** feature for the minority of studies that care about the
     process itself — association overhead, dynamic adaptation (B2).

3. **Do no harm.** Every existing 802.11 ≤ ac scenario must produce identical
   results unless it explicitly opts into a new feature/config. Backward-compat
   regression tests are part of "tests first."

4. **Reuse the structures already there** — the PHY mode hierarchy (`mode/`),
   `Ieee80211ModeSet` + `opMode`, the modular MAC (frame sequences, policies),
   the MIB, and the NIST/YANS error models are all sound extension points.

5. **Gate new advanced features behind flags, default off.** Beamforming and
   MU-MIMO build on the adopted abstracted gain/SINR MIMO model (§4) — a settled
   foundation, not an open decision.

## 2. Part A — Completing existing, incomplete features

These already exist in skeleton/partial form; we finish and fix them. Independent
of each other — can proceed in parallel. Effort/risk are rough.

### A1 — LDPC coding *(medium)*
Today: `// TODO LDPC codes` in `mode/Ieee80211VhtCode.h`; convolutional (BCC) only.
- **Tests first:** module test feeding a fixed SNR through the error model for the
  same MCS under BCC vs LDPC and asserting LDPC yields the expected lower PER;
  plus a full-sim fingerprint at a marginal SNR showing the goodput difference.
- **Then:** implement the LDPC coded path and add BER/PER curves to the NIST/YANS
  error models.

### A2 — TXOP & aggregation correctness *(low–medium; concrete bug fix)*
Today: `HtTxOpFs::selectHtTxOpSequence()` returns `0` (stub); VHT A-MPDU limits
not applied.
- **Tests first:** module test driving `HtTxOpFs` with a mock TXOP context and
  asserting it selects the correct frame sequence (not `0`) for each protected
  case; module test asserting VHT A-MPDU is bounded by the VHT maximum (well above
  HT). Full-sim TXOP scenario fingerprint.
- **Then:** implement the HT/VHT TXOP frame-exchange sequences (L-SIG / NAV-
  protected, dual-CTS, …) and the VHT aggregation/length bounds.

### A3 — PHY / bandwidth completeness *(medium–high)*
Today: 160 MHz present in the mode set but unexercised/untested; 80+80 MHz not
modeled; STBC hard-assumed off; GI hand-set.
- **Tests first:** parametric unit test asserting data rate **and** PPDU duration
  vs the standard's MCS tables for every BW (20/40/80/160/80+80) × NSS × GI ×
  coding combination; full-sim example per bandwidth.
- **Then:** promote **160 MHz** to a first-class tested config; add **80+80 MHz**
  as a bandwidth option; drive **GI** from the operating config (A→B1); *(optional)*
  model **STBC** instead of assuming it off.

### A4 — HT/VHT-aware rate adaptation *(medium)*
Today: only ARF/AARF/Onoe — legacy, not MCS/NSS/BW-aware (`mac/ratecontrol/`).
- **Tests first:** module test feeding the algorithm a scripted success/failure
  pattern and asserting it converges to the expected MCS×NSS; full-sim with a
  mobility/SNR sweep asserting it tracks the best rate.
- **Then:** add a Minstrel-HT-style algorithm over MCS × NSS × BW (keep the legacy
  ones for legacy modes); make `mac/rateselection/` honor the operating config and
  pick the best rate per link rather than a single fixed bitrate.

## 3. Part B — New WiFi 5 features

Entirely absent today. **B1 first** (foundation); B3/B4 depend on it and on §4;
B2 is optional and can land anytime.

### B1 — Operating-configuration (capability) model + direct control *(foundation; medium)*
The keystone — and per principle 2 it is about the **result**, not a handshake.
- **Tests first:** module test that sets the operating config directly (BW, NSS,
  GI, MCS set, coding, beamforming/MU flags) and asserts PHY+MAC adopt exactly it
  — no negotiation involved; full-sim with **heterogeneous** nodes whose
  configs are set independently, asserting each link uses the intended config.
- **Then:** introduce a first-class capability/operating-config object in the MIB,
  directly user-settable, consumed when selecting the operative `ModeSet` subset,
  bandwidth, NSS, GI, and coding. Replaces the need to hand-match `opMode`/BW/NSS
  on every node; provides the capability state that B3/B4 require. When unset,
  behavior is exactly today's.

### B2 — *(Optional)* simulated capability negotiation *(opt-in; medium)*
For studies that genuinely need the process (association cost, dynamic adaptation).
- **Tests first:** module test with mock peer IEs asserting the derived config
  equals the local∩peer intersection; **regression** test asserting that with
  negotiation enabled the resulting config — and thus the statistics — match the
  equivalent directly-set B1 config; full-sim association exchange.
- **Then:** add **HT/VHT Capabilities & Operation** Information Elements to the
  frame formats/serializers (beacon, probe, (re)assoc) and an **Operating Mode
  Notification** path; the handshake simply *produces* a B1 operating config.
  Off by default — selecting it never changes results vs. setting the same config
  directly.

### B3 — Transmit beamforming (VHT sounding) *(high; gated, off by default)*
- **Tests first:** module test asserting the modeled beamforming gain is applied
  to SINR/PER when enabled and absent when not; full-sim showing the expected
  goodput/range improvement on a beamformed link.
- **Then:** model the sounding exchange (NDP Announcement → NDP → compressed
  beamforming feedback) and apply an abstracted beamforming gain (§4). Behind a
  capability flag.

### B4 — DL MU-MIMO *(highest; gated, off by default)*
- **Tests first:** module test of the DL MU exchange and AP-side allocator with
  mock users, asserting concurrent per-user streams and correct multi-user
  acknowledgement; full-sim asserting the aggregate-throughput gain over
  single-user with the same airtime.
- **Then:** Group ID management, a DL multi-user PPDU, per-user feedback/ack, and a
  basic AP-side user/stream allocator, on top of the §4 MIMO abstraction. Behind a
  capability flag; off (and all ≤ ac legacy modes) behave exactly as today.

## 4. MIMO modeling — abstracted gain/SINR (adopted)

Beamforming and MU-MIMO (B3/B4) cannot sit on today's model, where **spatial
streams are only a data-rate multiplier** — there is no MIMO channel matrix, no
per-stream fading, no spatial-multiplexing capacity, no beamforming gain. This
plan **commits** to a single, deliberately lightweight abstraction rather than a
full channel model:

- **Beamforming** is an **SINR/gain bonus** applied to the affected link when
  sounding has occurred and beamforming is enabled — not a computed steering
  matrix.
- **DL MU-MIMO** is **concurrent per-user streams within one TXOP/PPDU**, each
  carrying its own MCS/NSS, with an **interference/cross-talk penalty** (an SINR
  degradation term) capturing imperfect spatial separation — not a real
  multi-user precoder.
- No channel matrix, correlation, or condition-number modeling.

Rationale: this stays at INET's existing abstraction altitude, is cheap to compute
and to validate, and is sufficient for the MAC-level studies these features serve
(airtime, scheduling, aggregate throughput). The gain and penalty are **scenario
parameters**, so studies calibrate them to whatever effective behavior they need.

The abstraction is **factored behind a small per-link effective-SINR adjustment**,
so a future statistical MIMO-channel model — or reuse by later multi-user work —
can replace the gain/penalty terms without touching the MAC. Each feature's "tests
first" encodes this model directly: B3's test asserts the beamforming gain is
applied to SINR/PER; B4's asserts concurrent streams plus the cross-talk penalty.

## 5. Cross-cutting & sequencing
- **Backward-compat regression** is part of every item's "tests first": legacy +
  simplified-mgmt paths must be byte-for-byte unchanged when no new config/flag is
  in play.
- **The operating config (B1) is the integration point**: bandwidth/NSS/GI/MCS/
  coding/beamforming/MU all flow from it, set directly (default) or by optional
  negotiation (B2) — never from hand-matched `opMode`.
- **Documentation last:** update `doc/src/users-guide/ch-80211.rst` to document
  11ac/VHT, the operating-config model, and remaining limitations.

**Suggested order:** Part A items in parallel (each independently shippable) →
B1 (foundation) → B3/B4 (gated, after §4) → B2 whenever a study needs it.
Examples that exercise 40/80/160/80+80 MHz, multiple NSS, short GI, beamforming,
and MU-MIMO double as the full-simulation tests and replace the 20-MHz-only
`examples/wireless/lan80211ac` config.
