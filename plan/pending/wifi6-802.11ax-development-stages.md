# WiFi 6 (802.11ax / HE) — Staged Development Plan

Staged roadmap for **delivering** WiFi 6 in INET: each stage interleaves *fixing
existing issues* in the current 802.11 model with *implementing the new HE
features*. The design rationale and principles live in the companion
[WiFi 6 design plan](wifi6-802.11ax.md) (composability over versioning,
parameterize don't proliferate, `opMode` selection); this document is the
*how/when* — ordering, milestones, and exit criteria.

Issue references below come from a scan of the current code and the user's WiFi 5
plans ([wifi5-802.11ac-current-model.md](wifi5-802.11ac-current-model.md),
[wifi5-802.11ac-completion-and-fixes.md](wifi5-802.11ac-completion-and-fixes.md));
Stage 0 is largely the subset of that WiFi 5 completion work that HE depends on.

Guiding constraint: every stage keeps existing WiFi ≤5 scenarios working; new
behavior stays behind new `opMode` values / capability flags.

---

## Stage 0 — Foundation fixes (prerequisites)

**Goal:** repair the existing-model gaps that would otherwise block or distort HE.
Mostly overlaps the WiFi 5 completion plan — land it first.

**Fix existing issues:**
- **LDPC coding** — `mode/Ieee80211VhtCode.h` / `Ieee80211HtCode.h` TODOs;
  convolutional-only today. HE mandates LDPC.
- **HT/VHT TXOP & aggregation** — `framesequence/HtTxOpFs.cc`
  `selectHtTxOpSequence()` returns 0 (stub); VHT A-MPDU max length not enforced.
- **Bandwidth & guard interval as data, not constants** —
  `mode/Ieee80211Ht/VhtMode` hard-set GI and subcarrier counts; generalize toward
  a configurable, RU-ready representation.
- **Operating-config / capability state** — bandwidth & NSS are hand-set NED
  params (`rateselection/RateSelection.ned`), not negotiated; introduce
  first-class capability state (foundation for HE Capabilities IE).
- **Subcarrier count abstraction** — `getNumberOfDataSubcarriers()` returns fixed
  literals per bandwidth; make it computable (precursor to per-RU mapping).

**Depends on:** WiFi 5 completion plan (Parts A1–A3, B1).
**Exit criteria:** LDPC selectable; HT/VHT TXOP/aggregation exercised by a test;
160 MHz path tested; bandwidth/GI/NSS expressible as configuration.

## Stage 1 — HE single-user PHY

**Goal:** an HE link that works *without* any multi-user features — new rates and
formats selectable via `opMode="ax"`.

**Implement (new):**
- `Ieee80211HeMode` by **extending the shared mode/data-mode/MCS structure** with
  HE parameters (per the design plan), not a cloned hierarchy.
- HE MCS tables + **1024-QAM**; LDPC wired in (from Stage 0).
- HE-SU (and HE-ER-SU) PPDU/preamble formats incl. HE-SIG.
- New numerology: 4× OFDM symbol, GIs 0.8/1.6/3.2 µs.
- 20/40/80/160 MHz via the generalized bandwidth model.
- Register the mode set; add `"ax"` / `"ax(6Ghz)"` to the `opMode @enum`.

**Depends on:** Stage 0.
**Exit criteria:** unit tests: HE SU data rates & PPDU durations match the
standard across MCS/NSS/BW/GI; an example network runs an HE SU link.

---

## Stage 2 — HE capability negotiation & MAC plumbing (still single-user)

**Goal:** HE-capable nodes discover each other and pick HE rates correctly — no
OFDMA yet, but the MAC stops assuming legacy-only.

**Fix existing issues:**
- **Capability storage in MIB** — `mib/Ieee80211Mib.h` holds only a `qos` bool;
  add per-peer capability fields.
- **Capability IE handling in mgmt** — `mgmt/Ieee80211MgmtAp.cc` /
  `Ieee80211MgmtSta.cc` TODOs (supportedRates, AID hardcoded to 0); parse/store
  HE Capabilities/Operation IEs; assign real AIDs.
- **Rate selection honoring basic/MCS sets** — `rateselection/RateSelection.cc`
  `BSSBasicRateSet` TODOs; filter by peer-supported HE MCS set.
- **Block Ack scale-up** — `blockack/QosAckHandler.cc` hard-coded 64-entry /
  16-fragment bitmap; widen toward 256-entry BA and drop the data-frame
  fragmentation assumption.
- **A-MSDU/A-MPDU limits** — `aggregation/BasicMsduAggregationPolicy.cc` fixed
  4065 B; raise to HE limits (A-MSDU 8191 B, larger A-MPDU).

**Implement (new):** HE Capabilities/Operation Information Elements; per-peer HE
capability record in the MIB used by rate selection.
**Depends on:** Stage 1.
**Exit criteria:** two HE nodes associate, exchange capabilities, and select an
HE MCS automatically (no hand-set rate); A-MPDU at HE limits.

## Stage 3 — OFDMA core (the centerpiece)

**Goal:** the defining WiFi 6 feature — multiple STAs share one PPDU via Resource
Units. This is the large structural change (single-peer → multi-peer MAC).

**Fix existing issues (structural):**
- **Multi-peer frame-sequence context** — `framesequence/FrameSequenceContext.h`
  and `queue/InProgressFrames.h` assume one frame/one peer; generalize to many
  STAs/RUs per exchange.
- **Multi-recipient Block Ack** — single `(addr, TID)` agreements and
  single-peer reorder buffers (`blockack/`, `blockackreordering/ReceiveBuffer`);
  add Multi-STA BlockAck (`QosAckHandler.cc` "Multi-TID BlockReq is
  unimplemented").
- **Per-user TXOP/rate state** — `originator/TxopProcedure`, `RateSelection`
  carry one peer/one mode; make per-RU/per-user.
- **Implement `McfFs`** / trigger-capable sequences (`framesequence/McfFs.cc`
  placeholder).

**Implement (new):**
- PHY: **Resource Unit** abstraction (26/52/106/242/484/996-tone), per-RU MCS,
  HE-MU & HE-TB PPDU formats; per-RU error/duration computation.
- MAC: **DL-OFDMA scheduler** (which STA → which RU); **Trigger frames** +
  **UL-OFDMA** synchronized trigger-based response sequences; **Multi-STA Block
  Ack**.

**Depends on:** Stage 2.
**Exit criteria:** an AP serves N STAs in one DL-OFDMA PPDU and solicits a
synchronized UL-OFDMA response acknowledged by a single Multi-STA BA; throughput
scales with users in an example scenario.

---

## Stage 4 — MU-MIMO & efficiency features

**Goal:** the remaining HE feature set, layered on the OFDMA machinery.

**Implement (new):**
- **DL/UL MU-MIMO** (spatial multiplexing, up to 8 users) reusing the multi-peer
  context and trigger framework.
- **BSS Coloring / spatial reuse** — color-aware deferral.
- **Target Wake Time (TWT)**.
- **6 GHz band operation (WiFi 6E)** channelization.

**Depends on:** Stage 3.
**Exit criteria:** MU-MIMO transmission to multiple STAs verified; spatial-reuse
scenario shows increased aggregate throughput; TWT schedule honored.

---

## Stage 5 — Hardening, examples, validation

**Goal:** make it trustworthy and demonstrable.

- New showcase/example networks for OFDMA, MU-MIMO, BSS coloring.
- Regression/fingerprint tests for all WiFi ≤5 scenarios (no behavioral drift).
- Throughput/latency sanity-checks against published 802.11ax expectations.
- Opportunistic cleanup surfaced during the work (FCS computation, protocol
  dissector A-MPDU parsing) as capacity allows.

**Depends on:** Stages 1–4.
**Exit criteria:** green regression suite; documented example results within
expected ranges of the standard.

---

### Stage dependency summary
`Stage 0 → 1 → 2 → 3 → 4 → 5`. Stage 3 (OFDMA) is the critical, highest-risk
stage and the prerequisite for everything WiFi 7 adds — see
[WiFi 7 staged plan](wifi7-802.11be-development-stages.md).
